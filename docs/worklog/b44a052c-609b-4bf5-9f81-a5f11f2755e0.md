**Approach A — 7-phase 순차 풀스윕** ⭐ 추천

1. Modal 통합 (사용자 강조): `dialog.tsx`로 단일화 + `ModalForm`/`ModalShell` 래퍼 추출 + `components/modals/`로 위치 이동
2. 안 쓰는 훅 채택: `useFormSubmit` / `useResourceList`를 18+ 파일에 적용
3. 컴포넌트 재사용: `ProviderRow`, `SectionCard`, `EmptyState` 등 중복 UI 패턴 추출
4. 큰 파일 분할: `lib/phoenix.ts`, `openapi-spec.ts`, `span-tree-view`, `prompt-builder`, `dataset-manager`, `trace-detail-tabs`, `chat-section`
5. 백엔드 API consistency 점검: 50개 route — 응답 포맷/에러 코드/validation/권한 체크 일관성
6. Dead code 2차: 최근 8일간 새로 생긴 미사용 export/파일
7. README + API docs 갱신

장점: 각 phase 독립 PR/commit, 리스크 낮음, 진행상황 가시화
단점: phase 수 많음 (실행은 길지만 안전)

**Approach B — Domain-vertical**
Traces/Datasets/Evals/Chat/Settings 도메인별로 UI+API+lib 한 번에 — 일관성은 좋지만 PR이 커서 리뷰/롤백 어려움

**Approach C — Top 5 핫스팟만**
가장 큰 파일 5개 + 모달 통합만. 나머지는 다음 라운드 — 빠르지만 사용자 요청 일부만 커버

**Approach A 추천 이유**: 사용자가 명시한 "모달 통합"을 Phase 1로 올리고, 풀스윕을 약속한 범위를 안전한 단위로 분할. 각 phase가 자체 verification 가능해서 테스트 인프라 부족을 보완.

## 설계 핵심 (Phase별)

### Phase 1 — Modal 통합
- **표준 선택**: `components/ui/dialog.tsx` (Radix 기반, focus trap·ESC·ARIA 자동) 채택. 커스텀 `modal.tsx`는 a11y 부족.
- **마이그레이션 부담 완화**: 기존 11곳이 쓰는 API 호환성을 위해 `ModalShell`을 Radix Dialog 위에 얇게 래핑 — `<ModalShell open onClose>...` 그대로 동작.
- **신규 `ModalForm`**: footer(저장/취소/loading/error) 표준화. `<ModalForm title onClose onSubmit saving error>...children...</ModalForm>` 한 줄로 폼 모달 완성.
- **위치 통일**: 인라인 모달(`projects/page.tsx`, `agents-section.tsx`, `dataset-manager.tsx`)과 `join-project-modal.tsx` → 모두 `components/modals/`로 이동.
- **종료**: `components/ui/modal.tsx` 삭제.

### Phase 2 — 만들어둔 훅 채택
- `useFormSubmit` → 18+ 파일 (agents/providers/general/chat-section + 모든 modal form)
- `useResourceList` → datasets/eval-prompts/agent-configs/threads 리스트
- 각 적용 시 try/catch/setSaving/setError 보일러플레이트 제거.

### Phase 3 — 재사용 컴포넌트 추출/확산
- 신규: `ProviderRow` (project settings ↔ global settings 거의 동일 → 통합)
- 기존이지만 채택 부족: `SectionCard`, `EmptyState`, `FormField` → 사용처 확대
- 카드/리스트 row 공통 패턴 추출

### Phase 4 — 큰 파일 분할
| 파일 | 분할 |
|---|---|
| `lib/phoenix.ts` (834) | `phoenix/{types,traces,prompts,llm,guardrail,index}.ts` |
| `lib/openapi-spec.ts` (778) | `lib/openapi/{index,projects,datasets,providers,evals,auth}.ts` |
| `span-tree-view.tsx` (767) | view + `span-tree-node` + `span-style` + helpers |
| `prompt-builder.tsx` (681) | builder + step components |
| `dataset-manager.tsx` (662) | orchestrator + sidebar + toolbar |
| `trace-detail-tabs.tsx` (616) | per-tab 파일 분리 |
| `chat-section.tsx` (618) | config 하위 섹션 분리 |

`@/lib/phoenix` import 경로는 barrel(`index.ts`)로 유지 → 외부 호출자 영향 없음.

### Phase 5 — 백엔드 API consistency
50개 route 감사 + 자동화 가능한 부분은 lint:
- 모두 `authedHandler` 사용 (현재 100% — 회귀 방지)
- 에러: `apiError(req, ErrorCode.X)` 강제 (raw `NextResponse.json({error})` 잔여 제거)
- project-scoped는 `requireProjectMember` 사용
- 응답 포맷 통일: `{ data }` vs raw 혼재 → 한쪽으로 통일
- 페이지네이션 파라미터 표준화 (`limit`, `cursor` or `page`/`pageSize`)
- 공통 listing 로직(필터+검색+페이지) `lib/api-helpers.ts`로 추출

### Phase 6 — Dead code 2차
- 미사용 export/파일 스캔 (knip 또는 grep)
- `.claude/worktrees/agent-*` (오래된 worktree 잔재) 정리 권유
- 최근 8일간 새로 생긴 dead spot

### Phase 7 — Docs
- `README.md` 갱신 (아키텍처/구조)
- `app/docs/sections/api.tsx` 50개 route 반영

### Phase 8 — 컨벤션 codification (사용자 요청)
- **`CLAUDE.md` 신규 작성/갱신**: 본 리팩토링 결과 컨벤션 명시
