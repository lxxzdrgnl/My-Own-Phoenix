## 1. Claude 훅이 뭐고, 파이썬으로 실행되는 거야?

훅은 **Claude Code 생애주기 이벤트마다 자동 실행되는 셸 명령**입니다. "파이썬 전용"이 아니라 그냥 셸 명령이고, 우리 건 그 명령이 마침 `python3 export-transcript.py`인 것뿐이에요. (bash든 node든 뭐든 가능)

동작 방식:
```
[이벤트 발생] → Claude Code가 명령 실행 → 명령의 stdin으로 JSON 전달 → 명령이 일하고 종료
```
우리 Stop 훅이 받는 stdin JSON:
```json
{ "session_id": "...", "transcript_path": "/home/rheon/.claude/projects/.../<id>.jsonl" }
```
스크립트는 그 `transcript_path`를 읽어 → `docs/worklog/<session-id>.md`로 변환 저장합니다.

**이벤트 종류** (우리가 쓴 건 `Stop`): `Stop`(매 응답 종료), `PreToolUse`/`PostToolUse`(도구 전후 — 이 프로젝트 하네스가 이미 사용 중), `SessionStart`, `PreCompact` 등.

**왜 토큰 0 / 느려짐 0인가:** 훅 출력이 stdout으로 안 나가면 모델 컨텍스트에 안 들어가서 토큰 0. 게다가 `"async": true`라 Claude Code가 백그라운드로 던지고 턴을 기다리지 않습니다 (그래서 `&`도 안 붙였어요 — `&`는 stdin이 닫히는 레이스 위험).

## 2. 세션 여러 개 + 서브에이전트에서도 되나?

**멀티 세션 → 완벽히 됨.**
세션마다 고유한 `transcript_path`(고유 session-id)를 가집니다. 훅은 각 세션에서 따로 fire하고, 그 세션의 파일을 → `docs/worklog/<해당-session-id>.md`로 씁니다. **세션당 md 1개**라 동시에 3개를 열어도 서로 다른 파일이라 충돌 없음.

**서브에이전트 → 주의점 1개:**
- 서브에이전트 종료는 `Stop`이 아니라 별도 이벤트 `SubagentStop`입니다. 지금 설정엔 `Stop`만 있어서 **서브에이전트 *내부* 단계는 worklog에 따로 안 찍힙니다.**
- 게다가 변환기가 `isSidechain`(서브에이전트 대화) 메시지를 **일부러 건너뜁니다.**
- **결과:** worklog엔 메인 대화 흐름(서브에이전트를 띄웠다는 사실 + 돌아온 최종 결과 포함)만 깔끔히 남고, 서브에이전트 내부 수다는 빠집니다 — 블로그 작업기록엔 오히려 이게 더 읽기 좋아요.
- 만약 서브에이전트 내부까지 다 남기고 싶으면 → `SubagentStop` 훅 추가 + sidechain 필터 해제하면 됩니다. 원하면 해드릴게요.

## 🔌 블로그 1 — Connector 적용기 (가장 강력, 추천)

**도입 이유 (코드로 확인됨):**
로컬에서 개발 중인 Dexter 에이전트를 **배포하지 않고**, 이미 배포된 운영 Phoenix(`phoenix.rheon.kr`)에 연결해 쓰기 위한 **WebSocket 릴레이**. 대시보드 → 릴레이 → 로컬 에이전트로 채팅 요청이 흐르고, 에이전트는 트레이스를 운영 Phoenix로 역방향 export. `pc_*` 키 → 서버가 sha256 해시 → `User.relayKeyHash` 매칭 → 프로젝트 멤버십 확인 → 릴레이 개통. 미니PC에 Docker로 상주.

**난관 → 해결 (핵심):**

| 증상 | 진짜 원인 | 해결 |
|---|---|---|
| WS 연결이 멈춤, `Cloudflare 524` | Cloudflare 프록시가 long-lived WS 업그레이드를 통과 못 시킴 | 프록시 우회 → 브리지 네트워크로 직접 (`ws://phoenix-dashboard:3001/...`) |
| 채팅이 "답을 안 하고 죽음", 404 | `--type=langgraph`인데 Dexter는 `/chat`만 노출 | `--type=rest`로 재시작 (→ `/chat` 라우팅) |
| "Connected to SaaS" 로그가 안 뜸 | Python이 파이프로 stdout 풀버퍼링 | `-e PYTHONUNBUFFERED=1`; 그리고 **로그 대신 DB `ConnectorSession` row로 생존 확인** |
| 채팅은 되는데 트레이스 안 잡힘, `ECONNREFUSED phoenix:6006` | dexter가 host 네트워크라 `phoenix` 호스트명 못 풂 | dexter+connector를 같은 브리지 네트워크(`servers_blueming-net`)로 |

**블로그 한 줄 교훈:** *"연결은 멀쩡했는데 로그가 거짓말을 했다."* — CLI stdout을 믿지 말고, 소켓 상태(`ss`/ESTABLISHED)와 서버 DB row를 ground truth로. 노트북을 운영에 꽂는 릴레이는 silent failure(프록시·프로토콜·버퍼링·Docker DNS)가 많으니 **실제 상태가 DB에서 관측되게** 설계.
제목 후보: *"로컬 에이전트를 배포 없이 운영 Phoenix에 꽂기 — Connector를 만든 이유와 4가지 함정"*

- **이번처럼 블로그(타 레포) 작업**: 오탐으로 마찰 발생.

**해결책 (택1):**

2. 블로그 작업 동안만 `PHOENIX_HARNESS=off`.
3. 그대로 두고 가끔 우회.
